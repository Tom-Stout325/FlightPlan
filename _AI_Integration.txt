You’d introduce three main pieces (conceptually):
WaiverRequest model
Stores all the form answers and the generated CONOPS text.
Fields like:
user, airspace_type, location, altitude, operation_dates, etc.
conops_draft (TextField)
conops_status (e.g., “not_generated”, “draft_created”, “edited”)
Waiver wizard view(s)
Could be a simple CBV or a multi-step wizard (like your incident report flow).
At the end, you have all the data validated and saved.
“Generate CONOPS” service
A Python helper function that:
Accepts a WaiverRequest instance (or cleaned data).
Builds the prompt.
Calls OpenAI.
Returns the CONOPS text and maybe some structured pieces (e.g., headings).
You keep all API logic server-side so:
Your API key is never exposed.
You can log/tune prompts centrally.
You can add rate limits per user.
4. Safety / policy / legal considerations
There are a few things you’ll want to bake into your design:
You’re not giving legal advice
Clearly label the output as:
“Draft CONOPS generated by AI for your review.”
“You are responsible for verifying accuracy and compliance before submitting.”
That keeps the relationship clear.
No “I have approval” language
Shape your system prompt so the model:
Describes proposed operations and mitigations.
Avoids implying FAA approval, certifications, or guarantees.
Don’t send unnecessary personal data
Only send to the API:
Operational parameters
Location, dates, altitudes
Aircraft type
Avoid names, phone numbers, addresses unless truly needed.
Auditability
Store:
The prompt inputs (or a redacted version)
The CONOPS draft as generated
That way the pilot (and you) have a record of what the AI produced.
